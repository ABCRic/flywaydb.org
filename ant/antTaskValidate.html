Ant Task: Validate

Validates the applied migrations against the ones available on the classpath. The build fails if differences in migration names, types or checksums are found.

Usage

<flyway:validate>
   <locations>
       <location path="com.mycompany.project.migration"/>
       <location path="database/migrations"/>
   </locations>
</flyway:validate>


Configuration

Attribute Ant / System Property Required Default Value Description
driver flyway.driver YES The fully qualified classname of the jdbc driver to use to connect to the database
url flyway.url YES The jdbc url to use to connect to the database
user flyway.user YES The user to use to connect to the database
password flyway.password NO blank The password to use to connect to the database
schemas flyway.schemas NO default schema of the connection Comma-separated list of schemas managed by Flyway. The first schema in the list will be the one containing the metadata table
table flyway.table NO schema_version The name of Flyway's metadata table.
By default (single-schema mode) the metadata table is placed in the default schema for the connection provided by the datasource.
When the flyway.schemas property is set (multi-schema mode), the metadata table is placed in the first schema of the list.
locations flyway.locations NO db/migration Locations on the classpath to scan recursively for migrations. Locations may contain both sql and java-based migrations.
Ant/System Property takes a comma-separated list.
sqlMigrationPrefix flyway.sqlMigrationPrefix NO V The file name prefix for Sql migrations
sqlMigrationSuffix flyway.sqlMigrationSuffix NO .sql The file name suffix for Sql migrations
encoding flyway.encoding NO UTF-8 The encoding of Sql migrations
validationErrorMode flyway.validationErrorMode NO FAIL The action to take when validation fails.

Possible values are:

FAIL
Throw an exception and fail

CLEAN (Warning ! Do not use in production !)
Cleans the database

This is exclusively intended as a convenience for development. Even tough we strongly recommend not to change migration scripts once they have been checked into SCM and run, this provides a way of dealing with this case in a smooth manner. The database will be wiped clean automatically, ensuring that the next migration will bring you back to the state checked into SCM.

This property has no effect when validationMode is set to NONE.
classpath flyway.classpath NO The Ant classpath used to load the JDBC driver and the migrations
classpathref flyway.classpathref NO The Ant classpath reference used to load the JDBC driver and the migrations


Sample configuration

<property name="flyway.driver" value="org.hsqldb.jdbcDriver"/>
<property name="flyway.url" value="jdbc:hsqldb:file:/db/flyway_sample"/>
<property name="flyway.user" value="SA"/>
<property name="flyway.password" value="mySecretPwd"/>
<property name="flyway.schemas" value="schema1,schema2,schema3"/>
<property name="flyway.table" value="schema_history"/>
<property name="flyway.locations" value="com.mycompany.project.migration,database/migrations"/>
<property name="flyway.sqlMigrationPrefix" value="Migration-"/>
<property name="flyway.sqlMigrationSuffix" value="-OK.sql"/>
<property name="flyway.encoding" value="ISO-8859-1"/>
<property name="flyway.validationErrorMode" value="FAIL"/>
